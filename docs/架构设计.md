## 2. Agent架构设计

### 2.1 Agent职责划分

#### 2.1.1 剧本解析Agent (ScriptParser)
**职责**：理解剧本结构，识别角色关系和情感线
**输入**：原始剧本文本
**输出**：结构化剧本数据

```python
{
  "title": "剧本名称",
  "characters": [
    {
      "name": "角色名",
      "relationships": [{"target": "另一角色", "type": "爱情/亲情/友情"}]
    }
  ],
  "episodes": [
    {
      "id": "ep_001",
      "name": "环节名称",
      "type": "narrative/interaction/clue/emotion",
      "content": "原内容",
      "position": 1
    }
  ]
}
```

#### 2.1.2 改编策略Agent (StrategyDesigner)
**职责**：分析角色特点，推荐合适的演绎类型和策略
**工作模式**：
- **自动生成模式**：分析环节 → 识别情感点 → 推荐演绎类型
- **协作模式**：评估DM想法 → 提出优化建议

**输出**：
```python
{
  "emotion_type": "爱情/亲情/友情/家国",
  "episode_type": "心建/羁绊仪式/情感爆发",
  "scene": "拜堂/对拜/递信物/回忆杀",
  "strategy": "通过仪式感增强情感投入",
  "reasoning": "该环节角色A与角色B即将分离，适合增加羁绊仪式"
}
```

#### 2.1.3 演绎设计Agent (PerformanceDesigner)
**职责**：将策略转化为完整的演绎手册
**输出格式**：
```python
{
  "title": "XX角色心建演绎方案",
  "duration": "5-8分钟",
  "props": ["红绸带", "信物", "背景音乐《XXX》"],
  "script": [
    {
      "step": 1,
      "action": "请玩家站起来，面向东方",
      "line": "（低沉）此刻，你站在你们初遇的地方...",
      "timing": "停顿3秒，让玩家进入情绪",
      "music": "播放《初见》前奏"
    }
  ],
  "notes": "注意玩家情绪，如过度激动需适当安抚"
}
```

#### 2.1.4 编排Agent (Orchestrator)
**职责**：任务调度、结果整合、对比分析
**核心功能**：
- 接收用户请求，创建任务流
- 协调各Agent执行顺序
- 整合多Agent输出
- 生成对比分析报告（双轨模式）

### 2.2 Agent协作流程

#### 2.2.1 标准流程（AI自动生成）
```
[用户] 上传剧本
   ↓
[编排Agent] 创建解析任务
   ↓
[剧本解析Agent] 解析剧本结构
   ↓
[用户] 选择目标环节 + 演绎维度（情感/环节/场景）
   ↓
[编排Agent] 创建设计任务
   ↓
[改编策略Agent] 分析并确定策略
   ↓
[演绎设计Agent] 生成完整演绎手册
   ↓
[编排Agent] 整合输出
   ↓
[用户] 查看、复制、导出
```

#### 2.2.2 双轨协作流程（DM提供想法）
```
[用户] 选择环节 + 提供改编想法
   ↓
[编排Agent] 创建双轨任务
   ├─→ [改编策略Agent] 自动生成策略
   └─→ [改编策略Agent] 评估DM想法
   ↓
[演绎设计Agent] 并行生成两个方案
   ↓
[编排Agent] 对比分析 → 优缺点对照表
   ↓
[用户] 选择方案 / 手动编辑
```

## 3. 数据模型设计

### 3.1 核心实体

```python
# 剧本
class Script:
    id: UUID
    title: str
    content: str              # 原始文本
    parsed_structure: dict    # 解析后的结构
    created_at: datetime
    updated_at: datetime

# 角色
class Character:
    id: UUID
    script_id: UUID
    name: str
    description: str
    relationships: list       # [{target_id, type}]

# 环节
class Episode:
    id: UUID
    script_id: UUID
    name: str
    content: str
    type: str                 # narrative/interaction/clue/emotion
    position: int

# 演绎方案
class PerformancePlan:
    id: UUID
    episode_id: UUID

    # 选择维度
    emotion_type: str         # 爱情/亲情/友情/家国
    episode_type: str         # 心建/羁绊仪式/情感爆发
    scene: str                # 具体场景

    # 生成内容
    title: str
    duration: str
    props: list               # 道具清单
    script: list              # 演绎脚本（动作+台词+节奏）
    notes: str                # 注意事项

    # 协作模式（可选）
    user_idea: str | None     # DM原始想法
    ai_plan: dict | None      # AI自动方案
    comparison: dict | None   # 对比分析

    created_at: datetime
```

## 4. API设计

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/v1/scripts` | POST | 创建剧本（文本） |

## 6. 关键设计决策

### 6.1 为什么选择多Agent架构？
- **单一职责**：每个Agent专注于特定任务，逻辑清晰
- **可替换性**：各Agent可独立优化或替换（如优化Prompt策略）
- **可扩展性**：未来可轻松增加新Agent（如音乐推荐Agent、道具管理Agent）

### 6.2 为什么输出完整手册而非实时指导？
- **DM使用习惯**：情感本DM需要提前熟悉流程，不能边看边带
- **仪式感要求**：仪式动作式演绎需要DM提前准备道具、设计走位
- **容错性**：现场突发情况时，DM需有完整把握才能灵活应对

### 6.3 大模型调用策略
直接调用Claude API，通过精心设计的Prompt实现功能。所有Agent通过Prompt Engineering完成各自任务，不计划进行模型微调。

## 8. 风险评估与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 大模型API成本过高 | 中 | 1. 实现智能缓存 2. 优化Prompt减少token 3. 限制单次生成长度 |
| 生成质量不稳定 | 高 | 1. 精细化Prompt工程 2. 多Agent校验机制 3. 用户反馈收集 |
| 剧本版权风险 | 中 | 1. 剧本本地存储不上云 2. 用户协议明确责任 3. 建议用户使用授权剧本 |
| 响应速度过慢 | 中 | 1. 异步任务队列 2. 流式响应 3. 优化Prompt减少token |
